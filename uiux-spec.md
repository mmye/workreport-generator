UI/UX Design Specification: Service Engineer Work Report Application
Document Overview
This document defines the UI/UX design and user flow for a work report application used by service engineers in the industrial machinery sector. The application provides a structured form-based report editor with two LLM-powered assistance features: Review (gap detection and improvement suggestions) and Polishing (readability enhancement). This document covers interaction design, layout, state management, and user flows. Technology and library choices are left to the implementer.


1. Core Design Principles
Principle
Description
Protect flow state
Never interrupt the writer mid-composition. All LLM feedback is triggered explicitly by the user.
Writer has final authority
Every LLM suggestion requires explicit acceptance. No silent auto-corrections.
Mode separation
"Writing" and "Reviewing/Polishing" are distinct modes the user consciously enters.
Minimal cognitive load
Service engineers return from field work fatigued. The UI must feel assistive, not supervisory.

4. Feature 2: Text Polishing (Readability Enhancement)
4.1 Purpose and Scope
Polishing improves readability without altering factual content. It is a separate function from Review and operates on different principles:

Review = discrete, card-based suggestions about content gaps and improvements
Polishing = continuous, field-level text transformation for readability
4.2 Recommended Workflow Position
Writing â†’ Review (content) â†’ Fix suggestions â†’ Polish (readability) â†’ Submit

Polishing is recommended after Review because content changes during review would invalidate polishing results. However, the UI does not enforce this order â€” only guides it (see Section 4.8).

8. Edge Cases and Special Behaviors
Scenario
Behavior
User manually edits a field after applying a suggestion
Applied status is retained. Changes are re-checked on next Review.
Multiple suggestions target the same field
Cards are displayed individually. If applying one invalidates another, dependent cards are auto-dismissed with a note.
Insert suggestion targets an empty field
Instead of a preview overlay, the proposed text is inserted in an editable draft state (pre-filled but not yet confirmed).
LLM suggestion is inappropriate
"Dismiss" with optional reason. Reason data can be collected for model improvement.
Re-review after polishing
Polished text is treated as the new baseline. Previous dismissals still apply.
User changes polishing level mid-preview
Previous polishing result is discarded. New diff is generated against the original (pre-polish) text.
Field has both active review suggestion and polishing preview
Review suggestions take precedence. Polishing preview is suppressed for that field until review suggestions are resolved.
